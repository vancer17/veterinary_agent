前置已完成：
1. 文本纠错、清洗得到干净语句；
2. 从 Redis 只读拉取当前会话**实体栈（LIFO，最新在前）**；
3. UIE / LTP 识别出：代词位置、代词类型（单数/复数/顺序/文件类/条件状态类）、代词修饰的业务类别；

最终消解一共 5 步：**候选实体过滤 → 优先级排序 → 歧义判定 → 原文字符串替换 → 消解结果归档**。

## 步骤1：按代词所属业务类型，过滤实体栈候选池（关键防错绑）
不能拿全栈所有实体去匹配，必须先按类型筛选，大幅降低歧义：
1. 人称代词、`它、该、此`：匹配设备、数据、任务结果类实体；
2. 文件类代词：这份表格、该附件、这个报表 → 只筛选 `excel_file / pdf_report / attachment` 类型实体；
3. 状态/条件代词：该筛选条件、上述规则、本次统计结果 → 只筛选 `filter_condition / stat_result`；
4. 顺序代词：前者、后者 → 不按类型过滤，直接取栈内有序位置。

过滤后得到同类型候选实体列表，依然保持「最新在前」顺序。

### 两种工具的代词类型识别方式
1. **UIE-Nano**
通过 Schema 直接抽取：`代词类型：单数/复数/文件指代/条件指代/前者后者`，自带业务分类标签。
2. **LTP**
- 词性 `r` 判定为代词；
- 依靠依存句法找到代词修饰的中心名词，通过中心词归类：
  - 中心词：报表、文件 → 文件类代词；
  - 中心词：条件、规则、结果 → 状态类代词。

## 步骤2：候选实体优先级排序（就近优先原则）
候选列表默认已经是栈内从新到旧，再叠加两条排序规则：
1. 同类型下：**越晚出现（越靠近栈顶）优先级越高**；
2. 同位置下：置信度越高优先级越高。

## 步骤3：歧义判定（保守策略，工程最容易踩坑）
### 场景A：候选列表为空
- 无任何可绑定实体 → **禁止自动消解**；
- 行为：保留原代词不替换，标记本次指代未知，向上反问用户：请问你说的「XX」具体指代哪一项内容？

### 场景B：候选数量 == 1（唯一匹配，安全可自动消解）
直接选取该实体作为替换目标，进入字符串替换环节。

### 场景C：候选数量 ≥ 2（存在歧义，严禁自动选择）
1. 不做任何自动替换；
2. 把Top2~Top3候选实体整理成选项给到用户确认；
3. 用户选定后，再执行替换；用户放弃则保留原文。

### 场景D：顺序代词（前者 / 后者）
- 后者：候选列表第 0 位（栈顶最新）
- 前者：候选列表第 1 位（上一个）
若候选不足2个，直接判定歧义，反问确认。

## 步骤4：原文精准字符串替换（最终消解核心操作）
### 4.1 获取代词在原文的字符偏移
1. **UIE-Nano**：抽取结果直接返回代词 `start、end` 字符下标，直接使用；
2. **LTP**：需要根据分词索引映射回原始文本字符偏移，定位代词在原文起止位置。

### 4.2 执行替换规则
1. 单数代词：直接用 `entity_text` 替换 `[start:end]` 区间的代词；
   示例：`把它导出` → `把近7天设备告警数据导出`
2. 复数代词（它们、这些文件）：多个实体用规范分隔词拼接：
   `近7天设备告警数据、厂区温度异常数据表`
3. 禁止简单关键词替换，必须**按字符区间替换**，避免同词多处误替换。

### 4.3 替换后校验
- 不能改变原有语句语义、不能篡改用户指令；
- 若替换后语句不通顺，回滚，保留原句，标记消解失败。

## 步骤5：消解结果归档、上下文更新（不修改实体栈）
1. 生成三条埋点日志：
   - 原始文本、代词位置、原代词内容；
   - 绑定的实体ID、实体内容、实体类型；
   - 消解后完整语句、消解策略（UIE规则/LTP句法、是否自动消解/用户确认消解）；
2. **实体栈只读不变**：本次消解只是读取历史实体做替换，不会新增、修改、删除Redis里的实体栈；
3. 消解后的标准化语句向下游传递：
   - 作为 Mem0 检索最终 Query；
   - 用于子任务参数回填、意图路由、工具调用参数填充。


# 指代消解后三层校验实现方案
消解后校验核心目标：**防止代词替换错位、语义篡改、语句不通顺、业务逻辑冲突、越界错误**，避免错误消解后的语句流入 Mem0 检索、任务拆分、工具调用链路。
分为五层递进校验：**位置边界校验 → 语句语法通顺校验 → 语义一致性校验 → 业务冲突校验 → 格式埋点校验**，同时兼容 UIE-Nano、LTP 两种抽取方案。

## 一、第一层：字符位置边界校验（必做，防止替换越界、错位）
### 校验场景
1. UIE/LTP 给出的代词 `start/end` 超出原始文本总长度；
2. 多轮代词依次替换后，后续代词偏移未动态修正，导致替换错位；
3. 分词索引转字符偏移时 LTP 映射出错，起止下标颠倒、负数。

### 校验规则
1. 前置基础校验
```
0 ≤ start < end ≤ len(original_clean_text)
start 不能等于 end，不能为负数
```
2. 多代词顺序替换动态偏移修正
每完成一次字符串替换，要根据替换前后字符长度差，修正后面所有待替换代词的 `start/end`：
`偏移修正值 = 替换后字符串长度 - 原代词片段长度`
后续所有代词位置统一累加修正值，避免越界替换。
3. 兜底策略：任意位置不合法 → 放弃本次代词替换，保留原始语句，标记`位置校验失败`。

### 两种工具特有校验点
- **UIE-Nano**：直接使用原始字符偏移，只需校验区间合法性；
- **LTP**：必须校验「分词索引→原始字符偏移」映射是否有效，若分词匹配失败直接放弃替换。

## 二、第二层：语句语法通顺性校验（过滤生硬、不通顺的错误替换）
1. 去除冗余连续标点、多个空格、残缺句式；
2. 使用 `KenLM` 语言模型分别计算：消解前PPL、消解后PPL困惑度；
3. 判定规则：
   - 消解后PPL 远高于消解前（语句通顺度大幅下降）→ 判定替换失败；
   - 阈值：若 `new_ppl > old_ppl * 1.5`，放弃消解，回滚原文。

### 兜底规则
替换后出现残缺句式、语义断裂、重复堆砌名词，直接回滚原始文本。

## 三、第三层：语义一致性校验（最核心，防止篡改用户原始意图）
### 校验目标
确保消解只做**代词等价替换**，不能增删、篡改用户原有业务诉求、约束条件。

#### 1. 轻量语义相似度校验（BGE/本地Embedding）
分别对「原始纠错语句」「消解后语句」做向量，计算余弦相似度：
- 推荐阈值：`sim ≥ 0.85` 才判定语义一致；
- 低于阈值：说明替换篡改了用户意图，回滚原文，消解失败。

> 典型错误示例：
> 原文：导出报表
> 错误消解：查询近7天设备数据并导出报表（新增查询动作，属于语义篡改，直接拦截）

#### 2. LLM小批量语义校验（兜底，仅前层失败时调用）
固定Prompt做一致性校验，仅返回是否通过：
```
判断：消解后的语句是否只是对原文代词做等价替换，没有新增、删减、修改用户的业务意图？
原文：{origin_text}
消解后文本：{resolved_text}
仅返回：通过 / 不通过 + 简短原因
```

## 四、第四层：业务冲突校验（结合会话上下文+实体类型约束）

### 2. 会话话题冲突校验
1. 拉取当前会话话题向量，计算消解后语句与当前话题相似度；
2. 若相似度低于话题阈值，判定替换后脱离当前业务话题，属于错误绑定历史实体，回滚。

## 六、五级校验整体放行&失败策略
### 放行条件（全部通过才允许使用消解后文本）
1. 字符偏移位置合法；
2. 语句通顺度达标；
3. 前后语义相似度≥0.85，无意图篡改；
4. 实体类型匹配、未脱离当前会话话题；
5. 格式、长度、敏感内容合规。

### 任意一级校验失败统一兜底策略
1. 放弃本次代词替换，**保留用户原始纠错语句**向下游流转；
2. 标记本次指代消解状态：`resolved: False、fail_reason: 具体失败原因`；
3. Mem0 仍使用原始语句检索，同时适当调高相似度阈值，规避错误召回；
4. 全量埋点失败日志，用于迭代代词规则、优化UIE/LTP抽取阈值。

## 七、UIE 与 LTP 在校验环节的差异
1. **UIE-Nano**
原生字符偏移基本不会错位，重点放在「语义相似度+实体类型+通顺度」三层校验即可；
2. **LTP**
必须额外增加「分词下标→字符偏移映射校验」，极易出现下标错位，优先做位置校验，再走后续四层规则。
