# 系统组件设计规范文档 (System Component Design Specification)

## 1. 核心管控原则

为遏制文档内容漂移（Document Drift），确保组件设计文档的长期一致性与高可维护性，所有组件文档必须严格遵循以下原则：

1. **单一事实源（Single Source of Truth）：** 文档仅界定组件的功能定义、系统边界及交互契约。严禁跨越组件边界描述上下游逻辑。
2. **引用优于复制（Reference over Replication）：** 严禁在文档中冗余业务需求说明（PRD）、数据定义语言（DDL）、环境配置及测试用例。关联系统信息必须通过统一资源定位符（URL）外部引用。
3. **契约导向（Contract-Oriented）：** 核心聚焦于输入输出模型（I/O）、异常处理机制及外部系统依赖，规避内部实现细节的流水账式记录。
4. **代码协同（Code Synchronization）：** 高频变更的结构化数据（如 API 报文结构体）需交由 API 治理平台统一管控，文档仅保留访问索引。

## 2. 文档内容准入矩阵

在编制组件设计文档时，请对照以下矩阵进行内容裁剪：

| 模块类别 | 准入内容（必须包含） | 禁入内容（强制外部引用或丢弃） |
| --- | --- | --- |
| **业务背景** | 阐明组件解决的核心技术或业务诉求。 | 完整业务 PRD、产品原型、用户故事。 |
| **接口契约** | 通信协议标准（如 RPC/REST）、核心接口职责及签名。 | 报文全量字段字典、请求/响应长篇示例。 |
| **数据模型** | 存储引擎选型（如 MySQL/Redis）及核心领域聚合根。 | 物理建表 DDL、索引明细、全量数据字典。 |
| **部署架构** | 弹性伸缩策略、系统资源预估及可用性指标（SLA）。 | 运维部署脚本、CI/CD 流水线配置、操作手册。 |
| **质量保障** | 核心容灾降级演练链路及预案策略。 | 详尽的测试用例矩阵、自动化测试代码。 |

## 4. 架构评审红线检查单 (Review Checklist)

文档评审阶段，若触犯以下任一条款，即刻执行驳回重构流程：

* [ ] **信息冗余违规：** 包含可由代码仓库或 API 治理平台自动生成的大篇幅结构化文本。
* [ ] **职责越权违规：** 描述了超出该组件物理或逻辑边界的上下游系统业务细节。
* [ ] **配置硬编码违规：** 文档内嵌具体的网络地址（IP）、测试环境标识或密钥，未使用逻辑配置描述。
* [ ] **边界定义缺失：** 未明确罗列“非目标（Non-Goals）”，存在未来功能无序蔓延（Feature Creep）风险。
